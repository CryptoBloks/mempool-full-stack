# Architecture

## Vision

This project is a configurator-driven, self-hosted Bitcoin infrastructure platform. An interactive wizard generates a single configuration file (`node.conf`), which drives template-based generation of all service configs, Docker Compose files, firewall rules, and more.

The stack gives operators a turnkey deployment of:
- Mempool.space block explorer (multi-network: mainnet, signet, testnet)
- Bitcoin Core full node(s)
- Electrs Electrum server(s)
- Optional Alchemy-style Bitcoin RPC web endpoint
- MariaDB backend
- BTRFS snapshot-based backup/restore with S3 streaming
- Cloudflare Tunnel for optional zero-trust remote access

---

## Configuration Flow

```
wizard.sh → node.conf → generate-config.sh → docker-compose.yml
                                            → config/{network}/bitcoin.conf
                                            → config/{network}/electrs.toml
                                            → config/{network}/mempool-config.json
                                            → config/openresty/nginx.conf
                                            → config/openresty/jsonrpc-access.lua (optional)
                                            → config/openresty/api-keys.json (optional)
                                            → config/mariadb/init/01-init.sql
                                            → config/cloudflared/config.yml (optional)
                                            → config/ufw-rules.sh
```

**Single source of truth:** `node.conf` — a KEY=VALUE file generated by the wizard. The generator reads it and renders all service configs from templates in `config/templates/`. Re-running the wizard with an existing `node.conf` pre-fills previous values as defaults.

### Template System

Templates use `{{PLACEHOLDER}}` syntax. The generator (`generate-config.sh`) replaces placeholders using:
- **Single-line values:** Bash parameter expansion
- **Multi-line values:** `awk` substitution

Templates are the only config files tracked in git. All generated output is gitignored.

## Multi-Network Architecture

### How It Works

The official mempool.space site uses a single frontend that routes to multiple backend instances via nginx path prefixes. We replicate this architecture:

```
                        ┌──────────────────────┐
                        │   OpenResty Proxy     │
                        │                       │
                        │   /            → mainnet frontend + API
                        │   /signet/     → signet API
                        │   /testnet/    → testnet API
                        │   /api/v1/     → mainnet API
                        │   /signet/api/ → signet API
                        │   /v2/     → RPC gateway (optional)
                        └────────┬──────────────┘
                                 │ internal docker network
          ┌──────────────────────┼──────────────────────┐
          │                      │                      │
          ▼                      ▼                      ▼
   ┌─────────────┐      ┌──────────────┐      ┌──────────────┐
   │   MAINNET   │      │   SIGNET     │      │   TESTNET    │
   │             │      │              │      │              │
   │  bitcoind   │      │  bitcoind    │      │  bitcoind    │
   │  electrs    │      │  electrs     │      │  electrs     │
   │  mempool-api│      │  mempool-api │      │  mempool-api │
   └─────────────┘      └──────────────┘      └──────────────┘
          │                      │                      │
          └──────────────┬───────┴──────────────────────┘
                  ┌──────┴───────┐     ┌─────────┐
                  │  mempool-web │     │ mariadb  │
                  │  (shared)    │     │ (shared) │
                  └──────────────┘     └──────────┘
```

### Container Inventory

**Per enabled network** (one set per network selected in the wizard):

| Container | Service | Internal Ports |
|-----------|---------|---------------|
| `bitcoind-{network}` | Bitcoin Core full node | RPC (8332/38332/18332), P2P (8333/38333/18333) |
| `electrs-{network}` | Electrs indexer | 50001 |
| `mempool-api-{network}` | Mempool backend API | 8999 |

**Shared across all networks:**

| Container | Service | Ports |
|-----------|---------|-------|
| `mariadb` | MariaDB (separate database per network) | 3306 (internal) |
| `mempool-web` | Mempool frontend (static Angular SPA) | 8080 (internal) |
| `openresty` | Reverse proxy + RPC gateway | 80/443 + RPC port (external) |
| `cloudflared` | Cloudflare Tunnel (optional) | none (outbound only) |

### Why Shared Services

- **MariaDB:** The production mempool.space runs one MariaDB server with separate databases (`mempool`, `mempool_signet`, `mempool_testnet`). Simpler and uses fewer resources than per-network instances.
- **Frontend:** The Angular SPA handles all network routing client-side. Only one instance is needed.
- **OpenResty:** Acts as the single external surface. Routes API requests by path prefix to the correct backend.

### Dependency Chain

```
bitcoind-{net} ──→ electrs-{net} ──→ mempool-api-{net}
                                           │
                                    mariadb (service_healthy)

openresty ──→ mempool-web + mempool-api-{net}
```

Each service waits for its upstream dependency to pass health checks before starting.

## Docker Network

All containers run on the `mempool_net` bridge network (172.20.0.0/24).

### External Ports

| Port | Service | Notes |
|------|---------|-------|
| 80/443 | Web UI | All networks via path prefix |
| 3000 (default) | RPC Gateway | All networks via path prefix (optional) |
| 8333 | Bitcoin P2P (mainnet) | Direct passthrough |
| 38333 | Bitcoin P2P (signet) | Direct passthrough |
| 18333 | Bitcoin P2P (testnet) | Direct passthrough |

### Internal Only (Docker Bridge)

| Port | Service | Notes |
|------|---------|-------|
| 8332/38332/18332 | Bitcoin RPC | Per-network containers |
| 50001 | Electrs | Per-network containers |
| 3306 | MariaDB | Shared instance |
| 8999 | Mempool API | Per-network containers |
| 8080 | Mempool Web | Shared instance |

Per-network services are separate containers, so they can all use the same internal port numbers without conflict. Docker DNS resolves by container name (e.g., `electrs-mainnet:50001` vs `electrs-signet:50001`).

## Storage Layout

```
${STORAGE_PATH}/                         # Default: /data/mempool
├── mainnet/
│   ├── bitcoin/                         # Bitcoin Core blockchain + indexes
│   │   ├── blocks/
│   │   ├── chainstate/
│   │   └── indexes/                     # txindex data
│   ├── electrs/                         # Electrs index
│   └── mempool-cache/                   # Mempool backend cache
├── signet/
│   ├── bitcoin/
│   ├── electrs/
│   └── mempool-cache/
├── mariadb/                             # Shared MariaDB data
└── snapshots/                           # BTRFS snapshots
```

**Key principle:** No runtime data inside the container or the repo. Config files are bind-mounted `:ro` from the repo's `config/` directory. Data lives entirely under `${STORAGE_PATH}`.

### BTRFS Integration

BTRFS is recommended but not required. The wizard detects the filesystem type on `${STORAGE_PATH}` and sets `BTRFS_ENABLED` in `node.conf`. At runtime, the backup and snapshot scripts independently detect BTRFS via `stat` and adjust behavior:

**When BTRFS is available:**
- `full-backup.sh` creates read-only snapshots before uploading, allowing services to restart immediately (~30-60s downtime)
- `restore.sh` recreates data directories as BTRFS subvolumes after deletion
- `snapshot/create.sh` creates atomic, space-efficient copy-on-write snapshots

**When BTRFS is not available:**
- `full-backup.sh` uploads directly from live data directories (services stay stopped for the entire upload — hours for mainnet)
- `restore.sh` uses regular `mkdir -p` for data directories
- `snapshot/create.sh` exits with an error (snapshots require BTRFS)

**Important:** For BTRFS snapshots to work, data directories must be BTRFS subvolumes. If you are setting up on a BTRFS filesystem, create the data directories as subvolumes before first start:

```bash
# Example for mainnet (run once before first docker compose up)
sudo btrfs subvolume create /data/mempool/mainnet/bitcoin
sudo btrfs subvolume create /data/mempool/mainnet/electrs
sudo btrfs subvolume create /data/mempool/mainnet/mempool-cache
sudo btrfs subvolume create /data/mempool/mariadb
sudo chown -R 1000:1000 /data/mempool/mainnet/{bitcoin,electrs,mempool-cache}
sudo chown -R 999:999 /data/mempool/mariadb
```

## RPC Gateway

When `RPC_ENDPOINT_ENABLED=true`, OpenResty exposes a JSON-RPC endpoint with defense-in-depth security:

```
Internet
  │
  │ HTTPS (port 3000/443)
  ▼
┌─────────────────────────────────────────────┐
│  OpenResty (nginx + Lua)                    │
│                                             │
│  1. TLS termination                         │
│  2. API key validation (URL path or header) │
│  3. JSON-RPC body parsing (Lua/CJSON)       │
│  4. Method whitelist check                  │
│  5. Per-key rate limiting (lua-resty-limit) │
│  6. CORS headers                            │
│  7. Per-network routing                     │
└──────────────┬──────────────────────────────┘
               │ HTTP (internal docker network)
               ▼
┌─────────────────────────────────────────────┐
│  Bitcoin Core (internal only)               │
│                                             │
│  Defense-in-depth:                          │
│  - rpcwhitelist per RPC user                │
│  - rpcallowip=172.20.0.0/24 (docker only)  │
│  - rpcwhitelistdefault=0                    │
└─────────────────────────────────────────────┘
```

### Endpoint Formats

Both are supported:
- **Path-based:** `POST /v2/{api-key}[/{network}]`
- **Header-based:** `POST /v2/ + X-API-Key: {key}`

### Method Whitelist Profiles

| Profile | Description | Use Case |
|---------|-------------|----------|
| `read-only` | Safe read methods only | Public access |
| `standard` | Read + transaction broadcast | App integration |
| `full` | All methods except always-blocked | Admin use |

Always-blocked methods: `stop`, `dumpprivkey`, `dumpwallet`, `importprivkey`, `invalidateblock`, `pruneblockchain`, `generateblock`, `generatetoaddress`.

## Security Model

### Container Hardening

All containers enforce:
```yaml
security_opt:
  - no-new-privileges:true
cap_drop:
  - ALL
restart: unless-stopped
```

### Network Isolation

- Only OpenResty is externally accessible (web + RPC ports)
- Bitcoin P2P ports are passed through directly
- All inter-service communication stays on the Docker bridge
- RPC, Electrs, and MariaDB ports never leave the bridge network

### Firewall (UFW)

Generated `config/ufw-rules.sh` includes:
- Docker-aware rules using the DOCKER-USER iptables chain (prevents Docker from bypassing UFW)
- Tunnel-aware logic: when Cloudflare Tunnel is enabled, web/RPC ports are NOT opened publicly
- Bitcoin P2P ports are always open
- SSH is always allowed

### Credentials

- Auto-generated by the wizard using cryptographic random hex
- Stored in `node.conf` (gitignored)
- Separate RPC users for internal services (full access) and the gateway (whitelisted methods)
- All config files mounted into containers as read-only

## Shared Libraries

### scripts/lib/common.sh

Provides logging, interactive prompts, validators, and utilities:
- `log_info()`, `log_warn()`, `log_error()`, `log_success()`, `log_header()`
- `ask_yes_no()`, `ask_input()`, `ask_choice()`, `ask_secret()`
- `validate_ip()`, `validate_port()`, `validate_path()`, `validate_url()`
- `require_command()`, `generate_password()`

### scripts/lib/config-utils.sh

Node.conf read/write with an in-memory associative array:
- `load_config()`, `get_config()`, `set_config()`, `config_exists()`
- `get_networks()`, `is_network_enabled()`, `get_network_config()`

### scripts/lib/network-defaults.sh

Per-network defaults, chain parameters, and Docker image mappings:
- `validate_network()`, `get_default_ports()`, `get_bitcoin_network_flag()`
- `get_default_versions()`, `get_chain_params()`, `get_docker_image()`

## Design Decisions

1. **Electrs over Fulcrum as default:** Lighter, official `mempool/electrs` Docker image. Fulcrum Dockerfile provided for operators with beefy hardware. TODO comments mark future per-network indexer choice.

2. **Unified multi-network deployment:** One compose stack with per-network `bitcoind`, `electrs`, `mempool-api`. Shared `mariadb`, `mempool-web`, `openresty`. Network selection via multi-select (default: mainnet + signet).

3. **OpenResty for everything:** TLS termination, multi-network API routing, and RPC gateway Lua filtering — all in one container.

4. **MariaDB init idempotency:** `generate-config.sh` produces init SQL with `CREATE DATABASE IF NOT EXISTS`. `start.sh` applies it on every start so databases are created for networks added after initial setup.

5. **`mempool/bitcoin` as default image:** Pre-built from Docker Hub. TODO comment marks future image configurability for alternatives.
